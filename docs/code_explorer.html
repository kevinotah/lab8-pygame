What the program does

The app opens a window, creates a list of square objects, and repeatedly updates and draws them. Each square has its own size, color, position, angle, speed, lifespan, and flee response.

The simulation now uses time-based movement: instead of moving a fixed amount per frame, it multiplies velocity by dt, so the motion stays consistent when frame rate changes.

# frame timing dt = clock.tick(FPS) / 1000 square.update(squares, dt) square.x += vx * dt square.y += vy * dt

Current behavior summary

  • Direction changes happen on a time interval, not every frame.
  • Smaller squares can push away from larger nearby squares.
  • Squares bounce when they reach the screen edges.
  • An FPS label and square count are drawn over the scene.

This is still a compact teaching project, but it now demonstrates a few important game-loop ideas: state, time, force-like steering, and render separation.

Runtime flow

flowchart TD
  A[pygame.init()] --> B[Create window]
  B --> C[Build squares list]
  C --> D[Main loop]
  D --> E[Read events]
  E --> F[clock.tick(FPS) -> dt]
  F --> G[Update each square]
  G --> H[Compute flee vector]
  H --> I[Move + bounce]
  I --> J[Draw background]
  J --> K[Draw squares]
  K --> L[Draw FPS/count text]
  L --> M[pygame.display.flip()]
  M --> D
The loop is simple and predictable: event handling, update, render, flip, repeat.

Module dependencies

pygame handles windowing, drawing, fonts, and the clock. random supplies unpredictability. math handles angles, trigonometry, and distance.

Class boundary

Square owns one square’s data and behavior. That includes movement, flee calculation, and drawing. The main loop only coordinates the list of squares.

Why this is a good teaching shape

The code separates object state from world orchestration, which makes it easier to reason about what belongs inside the object and what belongs in the loop.

Square state machine

stateDiagram-v2
  [*] --> Spawned
  Spawned --> Active: random size/color/position
  Active --> Steering: dt advances timer
  Steering --> Active: angle changes
  Active --> Fleeing: nearby larger square detected
  Fleeing --> Active: threat fades or list changes
  Active --> Bounced: wall hit
  Bounced --> Active: reflected angle
  Active --> Expired: lifespan exceeded
  Expired --> Replaced: new Square() appended
  Replaced --> Spawned
The square is not formally modeled with states in code, but the behavior naturally moves through these phases.

Important fields

  • x, y: top-left screen position
  • size: width and height of the square
  • color: RGB fill color
  • angle: direction of motion in radians
  • direction_timer: counts elapsed seconds since the last direction change
  • lifespan and certain_variable: elapsed time until replacement

Why time-based motion matters

With dt, the simulation is less tied to frame rate. A fast machine and a slow machine both produce roughly the same movement speed.

Flee behavior

flowchart TD
  A[Square.update()] --> B[Loop over all_squares]
  B --> C{Is it self?}
  C -- yes --> B
  C -- no --> D[Compute dx, dy]
  D --> E[Compute distance]
              E --> F{Other square bigger and within danger_distance?}
  F -- no --> B
  F -- yes --> G[Normalize away vector]
  G --> H[Add to flee_dx/flee_dy]
  H --> B
  B --> I[Blend flee with base motion]
  I --> J[Move square]
The flee system is a steering rule: it does not override randomness completely, it biases it away from nearby threats.

How the flee vector works

For each square, the code checks every other square. If the other square is bigger and close enough, it adds a vector pointing away from that square. Those small pushes accumulate into one flee direction.

Why it still feels random

The flee vector is blended with the square’s base random velocity. That means the square still wanders, but danger bends the path instead of completely replacing it.

Practical tuning knobs

  • danger_distance controls reaction range.
  • The 200 multiplier controls flee strength.
  • Lowering the multiplier makes the motion subtler.
1
Mutating the list while iterating
The draw pass removes and appends squares while looping through squares. That can skip elements and make the replacement behavior inconsistent.
2
A misleading lifespan variable name
certain_variable looks temporary, but it really tracks elapsed seconds. A clearer name would make the code easier to teach and maintain.
3
Flee checks scale quadratically
The nested comparison against all other squares is fine here, but the cost grows quickly if the square count rises much higher.
4
Static FPS surface
The FPS label is rendered once and reused. That is efficient, but it shows the target FPS, not the measured runtime FPS.
1. Fix the list mutation pattern

Build a new list of survivors each frame instead of removing and appending while iterating over the live list.

2. Rename the elapsed-time field

Use a name like age_seconds or elapsed_time so the intent is obvious.

3. Separate movement into helpers

Keep update() small by splitting random steering, flee steering, bounce handling, and aging into separate methods if the project grows.

4. Consider using squared distance

Compare squared values in the flee check to avoid a square root on every pairwise comparison.

5. Add a dynamic FPS readout if needed

If you want to monitor performance, render the actual measured FPS each frame or update the label periodically.
