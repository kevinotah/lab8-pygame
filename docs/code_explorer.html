Potential Issues
⚠️Neighbor checks scale poorly
Every square scans every other square when computing avoidance, so the work grows quickly as the count increases.
BasicsOne square does a full pass through the list of all squares.
Engineering InsightThat becomes expensive once the scene is larger than the current 100 squares.
Architecture InsightA spatial grid or radius filter would reduce unnecessary comparisons.
Time complexity overview
⚠️Types stay implicit
The code is readable, but the function contracts and data shapes are only implied, not stated.
BasicsReaders must infer what each parameter and return value should be.
Engineering InsightThis makes refactors and debugging a bit less safe than they need to be.
Architecture InsightAdding annotations would document the object and frame data flow directly.

Strengths

One place for game state

The Square class keeps movement data together, so it will be straightforward to annotate later.

Simple function boundaries

The file already separates setup, update, drawing, and helper logic, which helps when adding annotations.

Gaps

Missing parameter types

update, compute_flee_vector, draw, and main all rely on implied argument types.

Missing return types

None of the functions say what they return, even though some clearly return tuples or nothing at all.

Concrete generics are absent

list[Square] and tuple[float, float] would make the frame data much easier to understand.

Risks
Quadratic avoidance check
Every square compares itself to every other square, so the neighbor work is O(n^2) per frame.
Repeated trig and distance math
The loop repeatedly calls sin, cos, and sqrt, which is fine at 100 squares but scales poorly.
